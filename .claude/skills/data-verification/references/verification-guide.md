# データ検証ガイド

## 検証項目の詳細と実装方法

### 1. 欠損足の検出

**考え方:** 時間軸ごとの間隔で「あるべきタイムスタンプ」を生成し、実データと突き合わせる。

**時間軸ごとの間隔（ミリ秒）:**

| 時間軸 | 間隔 (ms) |
|--------|----------|
| 1min   | 60,000 |
| 5min   | 300,000 |
| 15min  | 900,000 |
| 30min  | 1,800,000 |
| 1hour  | 3,600,000 |
| 4hour  | 14,400,000 |
| 8hour  | 28,800,000 |
| 12hour | 43,200,000 |
| 1day   | 86,400,000 |
| 1week  | 604,800,000 |
| 1month | 不定（月末日による） |

**手順:**
1. データの最初と最後のタイムスタンプを取得
2. 間隔で割って期待本数を計算: `期待本数 = (末尾 - 先頭) / 間隔 + 1`
3. 実際の本数と比較
4. 差分があれば、期待タイムスタンプ列を生成して欠損箇所を特定

**暗号資産の特性:**
- 24 時間 365 日取引。株式のような「正当な休場日」はない
- 欠損の原因: 取引所メンテナンス、API 障害、極端な低流動性
- `1month` は月ごとに日数が異なるため、間隔ベースのチェックは不向き。代わりに月単位の連続性を確認する

### 2. 重複タイムスタンプの検出

**手順:**
1. 全タイムスタンプを配列に抽出
2. Set に変換して元配列と長さを比較
3. 差分 > 0 なら重複あり
4. 重複タイムスタンプを特定して報告

**よくある原因:**
- ページネーション境界での重複取得（`--from/--to` 利用時）
- CLI のキャッシュ結合時の境界重複

### 3. OHLCV 整合性チェック

**ルール:**

```
各足について:
  high >= open   かつ   high >= close    （high は最高値）
  low  <= open   かつ   low  <= close    （low は最安値）
  high >= low                             （最高値 >= 最安値）
  volume >= 0                             （出来高は非負）
  open > 0, high > 0, low > 0, close > 0 （価格は正の値）
```

**違反時の解釈:**
- `high < close` → API 応答の不整合。そのまま分析すると指標が歪む
- `volume < 0` → 通常あり得ない。データ破損の可能性
- `price <= 0` → 明らかな異常データ

### 4. 異常値（外れ値）の検出

暗号資産はペアごとにボラティリティ特性が大きく異なるため、一律の閾値は不適切。
ペアカテゴリ（major / mid / minor）と時間軸で閾値を分ける。

**ペアカテゴリの分類:** `_shared/references/pair-classification.md` を参照。

bitbank 上の実際の 24h 出来高シェアに基づいて 3 段階に分類:

| カテゴリ | 基準 | 代表ペア |
|---------|------|---------|
| major | 出来高シェア 10% 以上 | btc_jpy (~30%), xrp_jpy (~25%), eth_jpy (~16%) |
| mid | 出来高シェア 1〜10% | doge_jpy (~7%), sol_jpy (~3%), ltc_jpy, ada_jpy 等 |
| minor | 出来高シェア 1% 未満 | mona_jpy, sand_jpy, gala_jpy, bcc_jpy 等 |

**デフォルト閾値テーブル:**

| | 短期足 (1min〜30min) | 中期足 (1hour〜12hour) | 長期足 (1day〜) |
|-------|---------------------|----------------------|----------------|
| major | 価格 ±3%, 出来高 30x | 価格 ±8%, 出来高 20x | 価格 ±15%, 出来高 15x |
| mid | 価格 ±5%, 出来高 30x | 価格 ±12%, 出来高 20x | 価格 ±20%, 出来高 15x |
| minor | 価格 ±8%, 出来高 50x | 価格 ±18%, 出来高 30x | 価格 ±30%, 出来高 20x |

**閾値設計の根拠:**
- major の短期足 ±3%: BTC が 1 分で 3% 動くのは flash crash レベル
- minor の中期足 ±18%: MONA 等は 1 時間で 15% 動くことがある
- 出来高は minor ほど日常的なスパイクが大きいため、閾値を緩める

**価格変動の計算:**
```
変動率 = (current_close - prev_close) / prev_close
フラグ条件: abs(変動率) > 閾値（ペアカテゴリ × 時間軸で決定）
```

**出来高変動の計算:**
```
倍率 = current_volume / prev_volume
フラグ条件: 倍率 > 閾値（prev_volume > 0 のときのみ）
```

**重要: 異常値の扱い方**

Vol.01（暗号資産データ分析入門）の知見:
- 暗号資産はファットテール分布を持ち、4σ 超の変動が珍しくない
- 統計的外れ値の機械的削除よりも、ドメイン知識に基づく判断を推奨
- フラグは「確認すべきイベント」として提示し、除去は推奨しない

例: 2024 年の BTC は ±4% 超の時間足変動が複数回発生（ETF 承認時等）。
これらは異常値ではなく、マーケットイベントとして分析に含めるべき。

### 5. 出来高ゼロ連続の検出

暗号資産、特に bitbank のアルトコインでは出来高ゼロの足が発生しうる。

**判定基準:**

| カテゴリ | フラグ条件 | 理由 |
|---------|----------|------|
| major | ゼロ出来高が 1 本でもあれば NG | BTC/ETH で出来高ゼロは取引所障害の可能性 |
| mid | 3 本以上連続でフラグ | 散発的なゼロは低流動性時間帯（深夜等）で起こりうる |
| minor | 3 本以上連続でフラグ | 同上。ただし頻発する場合は流動性リスクとして報告 |

**報告内容:**
- ゼロ出来高の連続区間（開始〜終了タイムスタンプ）
- 連続本数
- major ペアの場合は「取引所障害の可能性あり」と注記

### 6. ヒゲ異常（wick anomaly）の検出

清算カスケードやフラッシュクラッシュで、実体に対してヒゲだけが異常に長い足が出る。
OHLCV 整合性チェック（項目 3）は通るが、この足を含む指標計算は歪む可能性がある。

**計算:**
```
実体 = abs(close - open)
ヒゲ全体 = high - low

close == open（十字線）の場合:
  wick_ratio は計算不可 → ヒゲ全体の絶対値で評価
  フラグ条件: (high - low) > 直近 20 本の平均レンジの 5 倍

close != open の場合:
  wick_ratio = ヒゲ全体 / 実体
  フラグ条件: wick_ratio > 10
```

**暗号資産特有の背景:**
- レバレッジ取引の強制清算（ロスカット）が連鎖すると、一瞬だけ価格が飛ぶ
- bitbank はスポット取引所だが、他取引所の清算がアービトラージ経由で波及する
- ヒゲ異常の足は SMA・BB 等の計算を大きく歪めるため、フラグの価値が高い

### 7. 時系列連続性チェック

**手順:**
1. 隣接するタイムスタンプの差分を計算
2. 期待間隔と比較
3. 間隔が期待値と異なる箇所を検出

**欠損足チェック（項目 1）との関係:**
- 連続性チェックで不連続が見つかった場合、欠損足チェックの結果と照合
- 両方で検出 → 欠損確定
- 連続性のみで検出 → タイムスタンプの並び順の問題（ソート不備等）

## パラメータのカスタマイズ

ユーザーは 2 つの方法でパラメータを変更できる:

### 方法 1: 発話で指定（セッション内）
```
「閾値を 5% にして BTC のデータを検証して」
「出来高の閾値は 50 倍で見て」
```

### 方法 2: SKILL.md を直接編集（永続的）
`SKILL.md` の「カスタマイズ可能なパラメータ」セクションのデフォルト値を書き換える。
ローカル環境なので自由に編集して OK。

## 検証結果の活用

- **全項目 OK** → そのまま indicator-analysis / backtest に進んでよい
- **欠損足あり** → 欠損が分析に影響するか判断。短期間の欠損なら前方補完（ffill）で対処可能
- **OHLCV 違反あり** → 該当足を除外するか、別日のデータで再取得を検討
- **異常値フラグ** → マーケットイベントとの照合を推奨。除去は非推奨
- **出来高ゼロ連続** → major ペアなら取引所障害を疑う。minor ペアなら流動性リスクとして認識
- **ヒゲ異常** → その足を含むテクニカル指標（特に BB, SMA）が歪む可能性を認識。除去ではなく、指標解釈時の注意点として伝える
